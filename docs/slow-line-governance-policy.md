# 慢线路治理规则（V1）

日期：2026-07-02  
时区：Asia/Shanghai

## 1. 目标

这份规则用于回答一个具体问题：

当某条上游线路没有直接报错，但持续表现更慢时，系统应如何处理，才能同时兼顾：

1. 可用性
2. 首 Token 延迟
3. 缓存率

原则是：

`慢线路不等于坏线路；先降权，后隔离；先看窗口统计，不看单次样本。`

## 2. 当前代码现状

当前调度链路已经具备这些基础能力：

1. 运行态信号
   - `errorRate EWMA`
   - `TTFT EWMA`

2. 粘性逃逸
   - sticky 账号健康度变差时，可触发 `sticky_escape`
   - 当前全局默认阈值：
     - `sticky_escape_ttft_ms = 15000`
     - `sticky_escape_error_rate = 0.5`

3. 评分选路
   - `priority`
   - `load`
   - `queue`
   - `errorRate`
   - `TTFT`
   - `reset`

4. TopK 候选
   - 候选先评分，再进入 `TopK`
   - 之后再做加权选择顺序

5. 硬故障处理
   - `401 / 403 / 429 / 529`
   - `response.failed`
   - transport error
   - timeout
   - 可触发请求内 failover

## 3. 当前真实缺口

上游池模型和后台表单里，已经存在池级参数：

1. `sticky_escape_error_rate_threshold`
2. `sticky_escape_ttft_ms_threshold`

后台默认值也是：

1. `error_rate_threshold = 0.3`
2. `ttft_ms_threshold = 6000`

但当前实际选路里，sticky escape 读取的仍然是**全局配置默认值**，不是池级阈值。

也就是说，当前状态更准确的描述是：

1. 后台可以配置池级慢线路阈值
2. 这些字段已经入库、可展示
3. 但还没有完整进入实际调度判断

这属于必须补齐的实现缺口。

## 4. 慢线路治理的分层口径

后续应分成两层：

### 4.1 全局兜底阈值

用于：

1. 没有池绑定
2. 池策略缺失
3. 配置异常
4. 回退模式

建议保留当前偏宽松口径：

1. `TTFT >= 15000ms`
2. `errorRate >= 0.5`

### 4.2 池级业务阈值

用于真正的上游池精细调度。

建议默认口径：

1. `TTFT >= 6000ms`
2. `errorRate >= 0.3`

原因：

1. 这是当前后台默认值
2. 对缓存敏感池更合理
3. 比全局兜底更早发现“慢但未死”的线路

## 5. 治理动作分级

慢线路治理不要只有“正常 / 错误”两档，建议分为 4 档：

### A. 正常

条件：

1. `TTFT EWMA` 未超过阈值
2. `errorRate EWMA` 未超过阈值

动作：

1. 正常参与调度
2. 正常参与 sticky
3. 正常参与 `TopK`

### B. 轻度变慢

条件：

1. `TTFT EWMA` 超过阈值
2. 但没有明显错误率恶化
3. 也没有硬故障

动作：

1. 允许触发 `sticky_escape`
2. 评分层降低 TTFT 因子的竞争力
3. 不立即摘除
4. 不立即标错

解释：

这类线路可能只是近期慢，不一定真的坏。

### C. 明显劣化

条件：

1. `TTFT EWMA` 连续超阈值
2. 同时伴随错误率抬升，或持续排队过重
3. 在同池、同模型、相近负载下相对表现明显更差

动作：

1. 继续允许 failover 切走
2. 在评分层显著降权
3. 减少它进入 `TopK` 的概率
4. 保留少量恢复观察流量

解释：

这是“降权优先”的核心阶段，目标是少吃流量，但不是完全隔离。

### D. 硬故障 / 不可用

条件：

1. 明确上游报错
2. timeout / transport error
3. `rate_limit / overload / temp_unschedulable / status=error`

动作：

1. 当前请求立即 failover
2. 账号进入摘除或冷却
3. 恢复探针接管恢复判断

## 6. 样本与判断边界

慢线路治理必须避免这几种误判：

1. 因为一次大输入请求，把账号永久判慢
2. 因为一次偶发抖动，就频繁切号
3. 因为不同模型差异，把模型问题误判成线路问题

因此应遵守：

1. 优先使用 EWMA，不直接用单请求值
2. 只在同池内做相对比较
3. 尽量限定同模型或相近能力请求
4. 大输入、大上下文请求应弱化对慢线路判定的影响

## 7. 与缓存率的协同规则

慢线路治理如果太激进，会直接打烂缓存率。

因此必须加这 4 条限制：

1. 只对 sticky 做“逃逸”，不是对所有请求强制换号
2. 轻度变慢时先降权，不先摘除
3. 恢复后先少量回流，不立即抢主流量
4. `TopK` 维持小范围，避免一边降权一边把候选打得太散

## 8. 推荐默认参数

这是当前建议的默认策略，不代表全部已实现。

### 8.1 缓存敏感池

1. `TopK = 2`
2. `MaxFailoverHops = 3`
3. `StickyEscapeTTFT = 6000ms`
4. `StickyEscapeErrorRate = 0.3`

### 8.2 全局兜底

1. `TopK = 2`
2. `StickyEscapeTTFT = 15000ms`
3. `StickyEscapeErrorRate = 0.5`

### 8.3 解释

1. `6000ms`
   - 用于更早识别“线路还活着，但明显拖慢体验”

2. `15000ms`
   - 用于没有池策略时的宽松兜底

3. `0.3`
   - 更适合池内竞争环境

4. `0.5`
   - 更适合作为全局保险线

## 9. 建议实现顺序

后续如果开始做代码，建议严格按下面顺序：

1. 先把池级 `StickyEscapeTTFT / ErrorRate` 真正接入 `OpenAIRoutingPolicy`
2. 再让 sticky escape 优先读取池级阈值，缺失时才回退全局值
3. 再补“恢复后渐进回流”机制
4. 最后才做更激进的慢线路自动降权策略

## 10. 验收口径

只有下面 4 条都成立，才能说慢线路治理基本闭环：

1. 慢线路不会长期继续吃主流量
2. 正常线路不会因过度随机分流而失去缓存优势
3. 恢复线路会重新回到候选，但不会立即抢满流量
4. 管理台能看出阈值、当前状态、最近恢复情况
