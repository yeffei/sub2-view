# 缓存优先调度策略（V1）

日期：2026-07-02  
时区：Asia/Shanghai

## 1. 目的

这份规则用于约束上游池调度，不让“为了分流而分流”破坏缓存率。

目标不是单纯把流量打散，而是同时守住三件事：

1. 正常情况下尽量稳定命中同一账号
2. 故障时能快速切到可用账号
3. 切换后尽量不把缓存命中率打烂

## 2. 基本判断

高可用和高缓存率不是矛盾关系，但调度策略必须分场景：

1. 正常场景：稳定优先，缓存优先  
2. 故障场景：可用优先，快速切换  
3. 恢复场景：渐进恢复，避免刚恢复就抢满流量

如果正常场景也频繁换账号，缓存率一定会掉。

## 3. 核心原则

### 3.1 稳定命中优先于平均乱分

同一个分组、同一类请求、同一段会话内，应优先命中稳定账号。

不能为了“看起来更均匀”而每次随机打散候选顺序。

### 3.2 只有硬故障才触发切换

应触发 failover 的场景：

1. `401 / 403 / 429 / 529`
2. `response.failed`
3. transport error
4. timeout
5. 上游已被判定为 `rate_limit / overload / temp_unschedulable / status=error`

不应仅因一次轻微抖动、一次偶发慢样本就立刻切换账号。

### 3.3 有会话上下文的请求必须更保守

以下请求默认属于“缓存敏感请求”：

1. 带 `previous_response_id`
2. 命中 sticky account
3. 已绑定会话上下文
4. 明显依赖上游连续上下文的多轮请求

这类请求默认优先保持原账号，除非原账号已经不可用或触发明确 failover 条件。

### 3.4 恢复账号不能立刻满额回流

异常账号恢复后，应先回到“小流量观察”状态，再逐步恢复正常权重。

否则会出现：

1. 刚恢复就把稳定流量打散
2. 缓存上下文重新分裂
3. 恢复其实不稳定，但瞬间又吃满请求

## 4. 调度顺序约束

建议统一按下面顺序做选路：

1. `previous_response_id` 绑定优先
2. sticky account 优先
3. 池内稳定候选集选择
4. 只有命中 failover 条件时才换下一个成员

这意味着：

1. 会话连续性优先于普通负载均衡
2. 普通负载均衡优先于大范围洗牌
3. 大范围洗牌只能是兜底，不应成为常态

## 5. TopK 与分流约束

`TopK` 不是越大越好。

`TopK` 过大时，常见后果是：

1. 候选过多，命中分散
2. 会话更容易漂移
3. 上游缓存被稀释
4. 平均成功率可能没问题，但真实体验变差

### 5.1 默认建议

1. 缓存敏感池：`TopK = 1 ~ 2`
2. 纯吞吐优先池：`TopK = 2 ~ 3`
3. 没有充分观测证据前，不建议把 `TopK` 继续放大

### 5.2 解释

1. `TopK = 1`
   - 最稳
   - 缓存率最好
   - 但容易把流量集中到单成员

2. `TopK = 2`
   - 是当前最平衡的区间
   - 既保留一定分流能力，也不至于把缓存打得太散

3. `TopK >= 3`
   - 必须建立在更强观测基础上
   - 否则很容易出现“调度看起来聪明，缓存实际掉很多”

## 6. 慢线路治理边界

“慢”不能简单等于“坏”。

需要区分两类情况：

1. 真故障
   - 明确报错
   - 超时
   - 连续 `response.failed`
   - 连续 transport error

2. 只是当前请求本身很重
   - 输入很长
   - 推理任务更复杂
   - 上下文很大
   - 不是账号本身质量差

因此慢线路治理不能按单次样本立即切号，而应遵守：

1. 只看同池、同模型、相近负载下的相对表现
2. 用窗口统计，不用单点样本
3. 先降权，不先摘除
4. 只有明确恶化到影响稳定时，才进入隔离

## 7. 恢复探针与缓存率的关系

恢复探针的目标是“把真正恢复的成员重新纳入调度”，不是“持续扰动正常流量”。

因此探针应遵守：

1. 正常成员不高频探测
2. 只针对异常或可恢复成员探测
3. 探测请求使用最小测试单元
4. 恢复成功后先回到观察态，而不是直接抢主流量

## 8. 当前代码现状与本策略的关系

### 8.1 已经基本符合的部分

1. 上游硬故障会触发请求内 failover
2. 已经写出流内容后不会再切，避免流拼接损坏
3. sticky / previous response / pool routing 已经进入链路
4. 异常成员已有摘除与恢复探针机制

### 8.2 还需要继续收口的部分

1. 缓存优先的稳定分流规则，需要更明确的排序约束
2. 慢线路降权规则，还没有完全定稿
3. 恢复成员的渐进回流策略，还需要单独实现或增强
4. `TopK` 与缓存率的联动，还缺稳定观测面板和回归验证

## 9. 后续实现要求

后面凡是继续改上游池调度，都应满足下面 4 条：

1. 正常不乱切  
2. 故障能快切  
3. 恢复不抢流量  
4. 缓存率不因“平均分流”被明显打散

如果某个调度优化会明显降低缓存率，即使它能提升表面分流均匀度，也不应直接上线。

## 10. 推荐口径

后面对这条能力的内部要求，建议统一写成：

`故障时自动切换，正常时稳定命中，恢复时渐进回流，在保证可用性的同时尽量守住缓存率。`
